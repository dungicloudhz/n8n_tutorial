01 · Introduction

Dify AI Workflow
Toàn tập cho Developer

Dify là nền tảng open-source để xây dựng ứng dụng LLM (Large Language Model) production-ready. Với Workflow Engine, bạn có thể thiết kế pipeline AI phức tạp thông qua giao diện drag-and-drop — từ chatbot đơn giản đến hệ thống multi-agent phức tạp.

🔀
Visual Workflow
Xây dựng pipeline AI bằng drag-and-drop. Debug từng node với trace chi tiết.
🤖
Multi-LLM
GPT-4o, Claude, Gemini, Llama, Mistral — switch provider không cần đổi code.
📚
RAG tích hợp
Knowledge Base với vector search, hybrid retrieval, re-ranking built-in.
🔧
Tools & Plugins
Google Search, Slack, Notion, GitHub… hoặc tự viết custom tool qua HTTP/Python.
📡
API First
Publish workflow thành REST API ngay lập tức. Tích hợp vào app backend dễ dàng.
🏠
Self-hosted
Deploy trên Docker/K8s. Data ở lại infrastructure của bạn, full control.
💡 Dify vs LangChain vs n8n
Dify tập trung vào LLM application layer: visual workflow + RAG + multi-model. LangChain là Python framework (code-first). n8n là general workflow automation (không LLM-native). Dify là điểm giữa — visual như n8n nhưng LLM-first như LangChain.

Dify hỗ trợ Docker Compose (self-hosted) và cloud. Đối với developer, self-hosted là lựa chọn tốt nhất để có full control.

03 · Core Concepts

🧠 Khái niệm cốt lõi

Dify có 4 loại ứng dụng chính. Mỗi loại phù hợp với use case khác nhau.

💬
Chatbot
CONVERSATION TYPE
Chat single-turn hoặc multi-turn. Có memory (conversation history). Phù hợp với customer support, Q&A bot.
🔀
Workflow
PIPELINE TYPE
Automation pipeline với nhiều bước xử lý. Không có conversation history. Tốt cho data processing, content generation.
🤖
Chat Flow
HYBRID TYPE
Kết hợp workflow + conversation. Pipeline có context của cả cuộc hội thoại. Use case: AI assistant phức tạp.
🕵️
Agent
AUTONOMOUS TYPE
LLM tự quyết định dùng tool nào, bao nhiêu bước. ReAct / Function Calling pattern. Tốt cho research, complex tasks.
📚
Knowledge Base
DATA LAYER
Upload document → chunk → embed → lưu vector DB. Dùng trong workflow qua Knowledge Retrieval node.
🔧
Tools
INTEGRATION LAYER
Built-in (Google Search, DALL-E, Jina) hoặc custom (HTTP API, Python function). Dùng trong Agent hoặc Tool node.
💡 Khi nào dùng loại nào?
Workflow: batch processing, scheduled jobs, deterministic pipeline (input → output cố định).
Chatbot/Chat Flow: conversation với users, cần nhớ lịch sử.
Agent: task phức tạp, không biết trước cần bao nhiêu bước, cần tự quyết định dùng tool.

05 · Nodes

🧩 Các loại Node

Dify cung cấp nhiều loại node sẵn có. Hiểu rõ từng loại là chìa khóa để thiết kế workflow hiệu quả.

Start
Entry Point
Điểm bắt đầu workflow. Định nghĩa input variables (String, Number, File, Select…). Mỗi workflow có đúng 1 Start node.
End
Exit Point
Kết thúc workflow và trả về output. Có thể có nhiều End node (trong các branch khác nhau).
LLM
AI Processing
Gọi LLM với system/user prompt. Hỗ trợ vision (ảnh), context injection, memory. Output: text, usage tokens.
IF/ELSE
Branching
Điều kiện rẽ nhánh. So sánh string, number, contains, is empty. Hỗ trợ AND/OR conditions.
Knowledge Retrieval
RAG
Truy vấn Knowledge Base. Trả về top-K chunks liên quan nhất theo query. Cấu hình similarity threshold, reranking.
Code
Custom Logic
Viết Python hoặc JavaScript tùy ý. Transform data, tính toán, gọi library. Sandbox isolated.
HTTP Request
Integration
Gọi bất kỳ REST API nào. GET/POST/PUT/DELETE. Headers, auth, body, timeout. Parse JSON response.
Template Transform
Data Transform
Jinja2 template engine. Kết hợp nhiều variables thành string. Dùng để format prompt, build JSON payload.
Iteration
Loop
Lặp qua array, xử lý từng item song song hoặc tuần tự. Sub-workflow bên trong. Tương tự forEach/map.

Conversation Variables (ChatFlow)

Chỉ available trong Chat Flow type. Tồn tại suốt cả cuộc hội thoại (persisted). Dùng để nhớ state giữa các lượt chat.

Code Node — Python 
# Code node: lưu language preference vào conversation variable
def main(user_message: str) -> dict:
    # Detect language từ message
    lang = "vi" if any(c > '\u0100' for c in user_message) else "en"
    return {
        "language": lang
    }

# Sau đó trong Variable Assigner node:
# conversation.user_language = code1.language
💡 Variable Aggregator
Khi workflow có IF/ELSE branch, mỗi nhánh có output riêng. Dùng Variable Aggregator node để merge output từ nhiều branch thành một variable duy nhất trước khi tiếp tục pipeline.
07 · Branching

🌿 Branching & Conditions

IF/ELSE node cho phép workflow rẽ nhánh dựa trên điều kiện. Đây là building block quan trọng để xây dựng logic phức tạp.

Cấu hình IF/ELSE Node

🟰
Toán tử so sánh
contains · not contains
starts with · ends with
is · is not
is empty · is not empty
= > < >= <= (number)
🔗
Logic operators
Kết hợp nhiều điều kiện:
AND — tất cả đúng
OR — ít nhất một đúng

Tạo nhiều ELIF branches cho multi-case.

Ví dụ: Route theo intent

▶ Start
user_query
→
🤖 LLM Classifier
intent detection
→
🔀 IF/ELSE
intent check
✅ TECHNICAL branch
Điều kiện: llm1.text contains "technical"

→ Tiếp theo: Knowledge Retrieval (tech docs) → LLM trả lời kỹ thuật
📋 ELSE branch
Fallback mặc định khi không match condition

→ Tiếp theo: LLM general response → End
LLM Classifier — System Prompt 
Phân loại câu hỏi sau vào một trong các nhóm:
- "technical": câu hỏi kỹ thuật, lập trình, API, lỗi
- "billing": câu hỏi thanh toán, giá, hoá đơn
- "general": câu hỏi chung, không thuộc nhóm trên

Chỉ trả về đúng một từ: technical | billing | general

Câu hỏi: {{#start.user_query#}}
⚠️ Lưu ý về Variable Aggregator
Sau IF/ELSE, mỗi branch tạo ra variable khác nhau (ví dụ: llm_tech.text vs llm_general.text). Phải dùng Variable Aggregator node để merge thành một biến final_answer trước End node. Nếu không, workflow sẽ lỗi.

Iteration node cho phép xử lý một mảng (array) dữ liệu bằng cách lặp qua từng phần tử, áp dụng sub-workflow bên trong.

→

trong sub-flow

Thuộc tínhMô tả
Input VariableBiến array đầu vào (Array[String], Array[Object]…)
itemPhần tử hiện tại trong vòng lặp — tham chiếu: {{#iteration.item#}}
indexIndex hiện tại (0-based) — tham chiếu: {{#iteration.index#}}
ParallelismSố lượng item xử lý song song (1 = tuần tự, max 10)

Ví dụ: Tóm tắt nhiều URLs

Thiết kế workflow 
Start node:
  - urls: Array[String]   # ["https://...", "https://..."]

Iteration node (input: start.urls):
  Sub-workflow bên trong:
  1. HTTP Request node:
     URL = {{#iteration.item#}}
     Method = GET
     
  2. Code node (Python):
     def main(html: str) -> dict:
         # Strip HTML tags, take first 2000 chars
         import re
         text = re.sub(r'<[^>]+>', '', html)[:2000]
         return {"clean_text": text}
     
  3. LLM node:
     Prompt: Tóm tắt ngắn gọn (2 câu): {{#code1.clean_text#}}

Sau Iteration:
  Template node: format array thành markdown list
  End node: trả về tổng hợp
💡 Parallel vs Sequential
Tăng Parallelism lên 5-10 để tăng tốc độ khi xử lý nhiều items. Giữ = 1 nếu cần duy trì thứ tự output hoặc có rate limit từ API bên ngoài.
09 · LLM Node

🤖 LLM Node — Chi tiết

LLM Node là trung tâm của mọi workflow. Hiểu sâu về cấu hình sẽ giúp bạn tạo ra output chính xác hơn.

🎛️
Model Settings
Temperature: 0.0 (deterministic) → 2.0 (creative). Dùng 0.0-0.3 cho extraction/classification, 0.7-1.0 cho content generation.
🔤
Context Window
Max tokens: giới hạn output length. Context: inject conversation history hoặc knowledge chunks vào prompt.
👁️
Vision
Với model hỗ trợ vision (GPT-4o, Claude 3), truyền sys.files vào LLM node để phân tích ảnh, PDF.
📤
Structured Output
Khai báo JSON Schema để LLM trả về JSON cấu trúc. Đảm bảo output parseable cho các node tiếp theo.

Prompt Engineering trong LLM Node

System Prompt — Best practice 
## Role
Bạn là AI assistant chuyên tư vấn sản phẩm cho {{#start.company_name#}}.

## Context từ Knowledge Base
{{#knowledge1.result[0].content#}}
{{#knowledge1.result[1].content#}}

## Rules
- Chỉ trả lời dựa trên context được cung cấp
- Nếu không biết, nói "Tôi không có thông tin về điều này"
- Trả lời bằng ngôn ngữ của user
- Tối đa 3 câu mỗi response

## Output format
Trả về JSON:
{
  "answer": "câu trả lời",
  "confidence": "high|medium|low",
  "sources": ["source1", "source2"]
}

Memory (Chat Flow)

Trong Chat Flow, enable Memory trong LLM node để inject conversation history vào context. Cấu hình window size (số lượt chat nhớ) và token limit.

⚠️ Context length gotcha
Knowledge chunks + conversation history + system prompt có thể nhanh chóng vượt quá context window của model. Theo dõi usage.total_tokens output của LLM node và cân nhắc dùng model context dài hơn hoặc giảm chunk size.
10 · Knowledge Base

📚 Knowledge Base — RAG

Knowledge Base là hệ thống RAG (Retrieval-Augmented Generation) tích hợp trong Dify. Upload document → tự động chunk → embed → lưu vector DB.

Tạo Knowledge Base

1
Tạo Knowledge
Menu trái → Knowledge → Create Knowledge → đặt tên.
2
Upload Document
Hỗ trợ PDF, TXT, MD, DOCX, HTML, CSV, JSON. Hoặc connect Notion, Website crawl, S3.
3
Cấu hình Chunking
Chọn strategy: Automatic (recommended cho bắt đầu) hoặc Custom (điều chỉnh chunk size, overlap).
4
Chọn Embedding Model
text-embedding-3-small (OpenAI, rẻ, nhanh) hoặc text-embedding-3-large (chính xác hơn). Dùng Ollama cho local embedding.

Retrieval Strategies

StrategyMô tảKhi nào dùng
Vector SearchTìm theo semantic similarity (cosine distance)Default, tốt nhất cho most cases
Full-text SearchKeyword matching (BM25)Tên riêng, số liệu, code snippets
Hybrid SearchKết hợp vector + keyword, rerankingProduction — accuracy cao nhất
✅ Hybrid Search + Reranking
Dùng Hybrid Search + bật Reranker (Cohere Rerank hoặc BGE Reranker) để có độ chính xác cao nhất. Reranker sắp xếp lại top-K chunks theo relevance thực sự, không chỉ theo vector distance.

Dùng trong Workflow

  Dựa trên thông tin sau để trả lời:
  ---
  {{#knowledge_retrieval.result[0].content#}}
  {{#knowledge_retrieval.result[1].content#}}
  {{#knowledge_retrieval.result[2].content#}}
  ---
  Nếu thông tin trên không đủ, hãy nói rõ.

User:
  {{#start.user_question#}}
11 · Agent

🕵️ Agent Node

Agent Node cho LLM tự động lên kế hoạch và quyết định dùng tool nào, bao nhiêu lần — thay vì pipeline cố định.

💡 Agent vs Workflow
Workflow: bạn thiết kế trước từng bước, deterministic. Agent: LLM tự quyết định flow, non-deterministic nhưng flexible hơn cho task phức tạp chưa biết trước số bước.

Agent Strategies

🔄
ReAct
Reason + Act cycle: LLM suy nghĩ → chọn tool → xem kết quả → lặp lại. Phổ biến nhất, hỗ trợ nhiều models.
⚡
Function Calling
OpenAI/Anthropic native tool use. Nhanh hơn ReAct, structured hơn. Cần model hỗ trợ function calling.

Thêm Tools cho Agent

  • Built-in Tools: Google Search, Bing, Wikipedia, DALL-E, Stable Diffusion, WolframAlpha, Jina Reader, GitHub, Notion, Slack, Telegram…
  • Custom Tools: Định nghĩa tool qua OpenAPI/Swagger spec. Dify tự generate tool interface từ schema.
  • Workflow as Tool: Biến một Workflow khác thành tool cho Agent gọi — kiến trúc multi-agent.
  • Knowledge Base: Attach knowledge base vào Agent để search document khi cần.
Custom Tool — OpenAPI Schema 
{
  "openapi": "3.0.0",
  "info": { "title": "Rider Status API", "version": "1.0" },
  "paths": {
    "/api/riders/{riderId}/status": {
      "get": {
        "summary": "Get rider contract status",
        "operationId": "getRiderStatus",
        "parameters": [{
          "name": "riderId",
          "in": "path",
          "required": true,
          "schema": { "type": "string" },
          "description": "Rider contract ID to check status"
        }]
      }
    }
  }
}
12 · Integration

🔧 Tools & HTTP Request Node

HTTP Request node là cách đơn giản nhất để tích hợp với bất kỳ external API nào — từ CRM, database API đến internal microservices.

Cấu hình HTTP Request Node

HTTP Request — POST to internal API 
Method: POST
URL:    https://api.internal.com/riders/{{#start.rider_id#}}/health

Headers:
  Content-Type: application/json
  Authorization: Bearer {{#env.API_TOKEN#}}
  X-Request-ID: {{#sys.workflow_run_id#}}

Body (JSON):
{
  "declaration": {{#start.health_data#}},
  "submittedAt": "{{#start.timestamp#}}"
}

Output variables:
  body        → JSON response
  status_code → HTTP status (200, 400, 500…)
  headers     → Response headers
⚠️ Environment Variables
Không hardcode API keys trong workflow. Dùng Environment Variables (Settings → Environment Variables) và tham chiếu bằng {{#env.VARIABLE_NAME#}}. Secrets được encrypted và không hiển thị trong logs.

Xử lý HTTP Response

Code Node — parse HTTP response 
def main(status_code: int, body: dict) -> dict:
    if status_code != 200:
        return {
            "success": False,
            "error": body.get("message", "Unknown error"),
            "data": None
        }
    
    return {
        "success": True,
        "error": None,
        "data": body.get("data", {})
    }
13 · Code Node

💻 Code Node

Code Node là công cụ mạnh nhất để thực hiện bất kỳ logic tùy ý nào — parse, transform, calculate, format data.

🐍 Python
  • Standard library đầy đủ
  • Built-in: json, re, math, datetime
  • Function signature cố định: def main(**kwargs)
  • Return phải là dict
📜 JavaScript
  • Node.js runtime
  • Async/await support
  • Entry point: async function main(args)
  • Return phải là plain object

Pattern hay dùng

Python — Xử lý JSON complex 
import json
import re
from datetime import datetime

def main(llm_output: str, user_id: str) -> dict:
    # Parse JSON từ LLM output (có thể có markdown fences)
    clean = re.sub(r'```json|```', '', llm_output).strip()
    
    try:
        data = json.loads(clean)
    except json.JSONDecodeError:
        return {
            "success": False,
            "error": "LLM không trả về JSON hợp lệ",
            "items": []
        }
    
    # Transform và enrich data
    items = data.get("items", [])
    enriched = [
        {**item, "user_id": user_id, "processed_at": datetime.now().isoformat()}
        for item in items
        if item.get("score", 0) > 0.5  # filter low confidence
    ]
    
    return {
        "success": True,
        "error": None,
        "items": enriched,
        "count": len(enriched)
    }
💡 Khai báo Input/Output
Trong Code node, bạn phải khai báo input variables (mapping từ node khác) và output variables (tên + type). Dify dùng khai báo này để validate và gợi ý trong node khác.
14 · Publish & API

📡 Publish Workflow thành API

Sau khi hoàn thiện workflow, publish nó thành REST API để tích hợp vào backend application — chỉ cần một API key.

Publish steps

Workflow editor → Publish (góc phải trên) → Access API → copy API Key và Endpoint URL.

Gọi Workflow API

Java / Spring Boot — Feign Client 
// Dify Workflow API — blocking (synchronous)
@FeignClient(name = "dify-client", url = "${dify.base-url}")
public interface DifyWorkflowClient {

    @PostMapping("/v1/workflows/run")
    DifyResponse runWorkflow(
        @RequestHeader("Authorization") String bearerToken,
        @RequestBody DifyRequest request
    );
}

// Request body
@Data
public class DifyRequest {
    private Map<String, Object> inputs;  // start node variables
    private String responseMode = "blocking";  // blocking | streaming
    private String user;  // unique user identifier
}

// Usage in Service
Map<String, Object> inputs = Map.of(
    "rider_id", riderId,
    "health_data", healthDeclaration
);

DifyResponse result = difyClient.runWorkflow(
    "Bearer " + apiKey,
    new DifyRequest(inputs, "blocking", userId)
);

15 · Debug

🐛 Debug & Tracing

Dify có hệ thống debug rất mạnh. Mỗi workflow run có trace đầy đủ từng node — input, output, latency, token usage.

Debug Panel

  • Node-by-node trace: Click vào từng node trong Run Log để xem exact input/output của node đó.
  • Token usage: Xem prompt_tokens, completion_tokens, total_tokens của mỗi LLM node. Tính cost.
  • Latency: Thời gian xử lý từng node. Identify bottleneck.
  • Error trace: Khi node fail, xem chính xác error message và stack trace.
  • Preview mode: Test workflow trong editor mà không cần publish. Nhập test inputs và chạy realtime.

Xử lý lỗi trong Workflow

🔴
Error Node
Mỗi node có thể cấu hình Error handling: fail workflow (default), hoặc tiếp tục với giá trị mặc định, hoặc route sang error branch.
🔄
Retry
HTTP Request node hỗ trợ Retry on error với max attempts và delay. Hữu ích cho flaky external APIs.

16 · Best Practices

📐 Best Practices & Patterns

Tổng hợp các pattern phổ biến và bài học thực tế khi xây dựng workflow production-ready.

Pattern 1: RAG với Fallback

Input
→
Knowledge Retrieval
→
IF score > 0.6
→
LLM (với context)

ELSE branch → LLM không có context → "Tôi không tìm thấy thông tin về câu hỏi này"

Pattern 2: Classification Router

Input
→
LLM Classifier
→
IF/ELSE (multi-branch)
→
Specialist Handlers
→
Variable Agg.
→
End

Pattern 3: Parallel Processing

Xử lý nhiều nguồn song song rồi merge. Ví dụ: tìm kiếm đồng thời từ Knowledge Base + Web Search + Database API.

❌ Sequential (chậm)
KB Search (1s) → Web Search (2s) → DB API (0.5s) → LLM (3s)
Total: ~6.5s
✅ Parallel + merge (nhanh)
[KB + Web + DB] chạy song song (max 2s) → Variable Agg. → LLM (3s)
Total: ~5s

Checklist Production Workflow

  • Environment Variables: Tất cả API keys, endpoints → vào Settings → Env Vars, không hardcode.
  • Error handling: Mọi HTTP Request node phải có error check (IF status_code != 200 → error branch).
  • Score threshold: Knowledge Retrieval nên có score_threshold ≥ 0.5 để tránh trả về chunks không liên quan.
  • Output validation: Code node validate JSON từ LLM trước khi dùng (LLM có thể hallucinate format).
  • Token budget: Set max_tokens phù hợp cho từng LLM node. Không để unlimited.
  • Logging: Dùng sys.workflow_run_id làm correlation ID khi gọi external API để trace end-to-end.
  • Test data: Lưu test cases (input + expected output) trong Dify Annotate để regression test khi update workflow.
  • Version control: Export workflow JSON thường xuyên và commit vào Git.
🎓 Tip cuối
Bắt đầu với workflow đơn giản nhất hoạt động được, sau đó iterate. Đừng cố thiết kế "perfect workflow" ngay từ đầu. Dify's visual editor giúp refactor nhanh — thêm branch, swap model, adjust prompt trong vài phút.
